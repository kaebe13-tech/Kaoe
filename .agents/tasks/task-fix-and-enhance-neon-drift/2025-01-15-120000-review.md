# Power-up suite, movement mechanics, and game-state fixes for Neon Drift

A batch of bug fixes and feature additions to a single-file Three.js endless runner. The fixes address a truncated hitbox call, missing HUD hidden style, broken building recycling, and a stale best-score element. The features add four new power-ups (Time Rewind, Magnet, Shield, and a reworked Time Stop), a slide/duck action, double jump, a score multiplier that ramps with distance, and speed-milestone notifications.

**Watch for:** Time Stop's speed restoration has an edge case where `deactivatePowerup()` is called on game-over while Time Stop is active, restoring the stored speed into a dead run — harmless only because the game transitions to GAME_OVER state first, but fragile if state transitions change (likely). The `checkCollisions` invincibility checks are duplicated between `checkCollisions` and `gameOver`, creating dead code where Glitch/TimeStop guards in `gameOver` are unreachable (confirmed). The Rewind checkpoint saves position at pickup time, but spawn cursors remain far ahead — after rewind the player is teleported backward into a region with stale obstacle layout and no new spawns, creating an unpredictable difficulty spike (confirmed).

---

## High-level view

Time Stop was rearchitected from a "freeze the world" mechanic into a slow-motion effect: speed drops to 25% and obstacles cannot kill. This introduces a stored-speed restoration pattern that couples the power-up lifecycle to game-over cleanup.

Rewind saves a spatial checkpoint on pickup and revives the player on next death. The teleportation clears nearby obstacles within 20 units, but the spawning cursors (`lastObstacleZ`, `lastTokenZ`, `lastPowerupZ`) remain far ahead — the region between the checkpoint and the old death point is already populated, so the player either replays the same obstacle layout or encounters token/power-up gaps.

Magnet pulls tokens toward the player each frame using distance-based linear falloff. Because it mutates `t.baseY` to track the attracted Y position, tokens that exit magnet range mid-pull keep their displaced height permanently.

Shield and Glitch provide competing invincibility semantics in a priority chain inside `gameOver()`: Glitch → Time Stop → Shield → Rewind. The same Glitch and Time Stop guards also appear inside `checkCollisions`, making the `gameOver` versions dead code — `checkCollisions` is the actual protection layer.

The slide mechanic shrinks the player hitbox vertically and skips laser collisions. `triggerRewind` resets `isSliding` and `slideTimer` but not `slideCooldown`, so a rewind during cooldown leaves the player unable to slide briefly after revival.

Score multiplier grows monotonically and is never capped — over a long run it inflates scores without bound.

## Time Stop rework and speed coupling

The new Time Stop multiplies speed by 0.25 and makes the player invincible to obstacles. The stored speed pattern:

```js
this.timeStopSpeedStore = this.speed;
this.speed = this.speed * 0.25;
// ...later...
this.speed = this.timeStopSpeedStore || CONFIG.BASE_SPEED;
```

The `|| CONFIG.BASE_SPEED` fallback silently masks bugs where `timeStopSpeedStore` is zero (e.g., if `deactivatePowerup` is called twice). `deactivatePowerup()` is called unconditionally in `gameOver()`. If a non-obstacle death path were ever added:

1. State → GAME_OVER
2. `deactivatePowerup()` → restores `this.speed` to the pre-slowdown value
3. Speed is now 4x what it was a frame ago — in a dead game state

Currently safe because no non-obstacle death exists. The `updateDifficulty` method skips speed increases during Time Stop, and `updateScore` uses the reduced speed — so distance-based multiplier growth also slows proportionally during slow-mo, preventing Time Stop from inflating scores.

## Rewind teleportation and world continuity

When the player dies with a rewind checkpoint:

```js
this.player.position.set(CONFIG.LANES[cp.lane], CONFIG.GROUND_Y, cp.z);
// Remove obstacles within 20 units of checkpoint
for (let i = this.obstacles.length - 1; i >= 0; i--) {
    if (Math.abs(o.mesh.position.z - cp.z) < 20) { ... remove ... }
}
```

Spawn cursors (`lastObstacleZ`, `lastTokenZ`, `lastPowerupZ`) are not touched. The `updateSpawning` condition:

```js
while (this.player.position.z - this.lastObstacleZ < CONFIG.SPAWN_AHEAD) this.advanceObstacleSpawn();
```

Since `this.player.position.z` is now `cp.z` (less negative) and `lastObstacleZ` is far more negative, the difference exceeds `SPAWN_AHEAD` → no new spawns happen. The player replays whatever obstacles remain between the checkpoint and where they died — potentially a long stretch with no tokens or power-ups, or a dense pre-existing obstacle field with no breathing room beyond the 20-unit clearance.

Fix: reset `lastObstacleZ = cp.z - 22` (or similar) so spawning resumes normally. Consider clearing obstacles/tokens/powerups beyond the checkpoint Z rather than just within 20 units.

## Collision double-guarding

`checkCollisions`:
```js
if (this.activePowerup === POWERUP_TYPE.TIMESTOP) continue;
if (this.activePowerup === POWERUP_TYPE.GLITCH) { particles; continue; }
if (this.isSliding && o.type === 'laser') continue;
this.gameOver();
```

`gameOver`:
```js
if (this.activePowerup === POWERUP_TYPE.GLITCH) return;
if (this.activePowerup === POWERUP_TYPE.TIMESTOP) return;
```

The `gameOver` guards are unreachable via the collision path. They'd only fire if `gameOver()` were called from another trigger — which doesn't exist. This creates ambiguity about where the invincibility policy lives. Recommend consolidating to one location.

## Slide-under-laser geometry

Lasers have collision offset Y=1.15, size Y=0.4 → box spans Y 0.95–1.35. A sliding player has hitbox center at `GROUND_Y - 0.3 = 0.3`, height 0.5 → box spans Y 0.05–0.55. These don't overlap geometrically, so the `isSliding && o.type === 'laser'` check is redundant with the spatial test. The type check remains valuable as a safety net if laser placement ever changes, but currently the geometry alone would suffice — meaning if the type check is accidentally removed, slides would still work against current lasers.

## Magnet token-pull displacement

```js
if (this.magnetActive) {
    const dist = t.mesh.position.distanceTo(this.player.position);
    if (dist < CONFIG.MAGNET_RADIUS) {
        const dir = tmpVec.copy(this.player.position).sub(t.mesh.position).normalize();
        const pullSpeed = 12 * (1 - dist / CONFIG.MAGNET_RADIUS);
        t.mesh.position.addScaledVector(dir, pullSpeed * dt);
        t.baseY = t.mesh.position.y;  // <-- permanent mutation
    }
}
```

If the magnet expires while a token is mid-pull but outside collection range, the token retains its displaced `baseY`. The 8-unit radius and 12-unit/s pull speed mean most tokens get collected before expiry, but a token at the outer edge that the player moves away from would remain at its displaced height.

## Dead field: `doubleJumpUsed`

Declared in the constructor and reset in `resetGame`, but actual double-jump logic uses `this.playerState.jumpCount`. Vestigial code from an earlier implementation approach.
